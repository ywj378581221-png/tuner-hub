# 从建站到上线：Tuner Hub 项目经验复盘

## 一、我们真正完成了什么

Tuner Hub 从一个“参考懂车帝布局的改装车分享网站”逐步变成了具备真实业务能力的网站：Vue 负责前台体验，Django 提供接口和管理后台，PostgreSQL 保存生产数据，Gunicorn 与 Nginx 在阿里云 ECS 上提供公网服务。

功能从最初的页面展示，扩展到了真实路由、注册登录、邮箱找回密码、用户资料、头像和昵称修改、关注与粉丝、私信、文章和动态发布、车型车友圈、车库、项目车记录、等级积分、账号封禁、内容管理、图片上传、文章分段排版、编辑以及回收站。

## 二、最重要的产品经验

### 1. 能点击不等于能使用

早期页面的问题不是“不好看”，而是很多按钮只改变了当前画面，没有真实路由、后端接口和数据库记录。真正的功能至少要满足：点击有效、数据保存、刷新后仍存在、其他用户能看到、权限正确。

### 2. 真实数据比热闹更重要

虚构的帖子、活动、车友会、粉丝数和关注数会让页面看起来丰富，却会伤害用户信任。社区冷启动阶段宁可诚实显示空状态，也不要把演示数据伪装成真实用户行为。

### 3. 从用户视角检查每个页面

判断一个页面是否合理，不能只看代码是否运行。还要问：用户为什么来到这里、最想看到什么、下一步要做什么、信息是否拥挤、按钮是否容易理解、失败后是否知道该怎么办。

### 4. 参考成熟产品要参考结构，不是照搬功能

懂车帝的内容布局、导航层级和阅读体验值得参考，但经销商报价、新车价格等能力与改装社区目标无关，应主动删除。产品功能必须服务自己的用户，而不是追求“看起来像大平台”。

## 三、前后端与数据库经验

### 1. 功能必须纵向打通

一个发布功能需要同时完成前端表单、文件选择、接口校验、数据库写入、权限判断、列表更新、详情展示和错误反馈。任何一层缺失，用户看到的都会是“按钮没用”或“发布后找不到”。

### 2. 路由必须真实

文章、帖子、车型、车友圈、关注、粉丝和回收站都应该有独立地址。这样刷新、分享链接、前进后退和直接访问才不会全部回到首页。

### 3. 图片不能依赖外站热链

直接引用大型汽车网站图片，可能因为防盗链、链接失效或访问限制而无法显示。更可靠的方式是由后台或用户上传图片，存入自己控制的媒体目录，并在前后端都提供验证和失败状态。

### 4. 文章需要结构化内容

长文章不能只保存为一整段文字。为了支持小标题、段落、正文图片、图片说明和任意插入位置，需要将正文保存成有顺序的内容块。编辑时还要保留已有图片，只有永久删除时才清理文件。

### 5. 权限必须由后端决定

隐藏前端按钮只能改善界面，不能阻止越权请求。文章编辑、删除、恢复、永久删除、置顶和封禁等操作，都必须在 Django 接口中再次检查作者、管理员和账号状态。

## 四、安全经验

- 密码使用 Django 的安全哈希保存，不能明文入库。
- 找回密码使用有时效的令牌，通过用户邮箱完成，并避免泄露某个邮箱是否注册。
- 密钥、数据库地址和邮箱授权码只保存在服务器环境文件中，不能提交到 GitHub。
- 后台、上传接口和管理操作都要验证登录状态与权限。
- 生产环境要正确配置 `ALLOWED_HOSTS`、CSRF、Cookie、安全组和 Nginx。
- 数据库和上传图片需要定期备份，永久删除前要明确提醒不可恢复。

## 五、上线与运维经验

一次完整更新不是只有 `git pull`。正确流程包括：加载生产环境、拉取代码、确认数据库、执行迁移、构建 Vue、收集 Django 静态文件、重启 Gunicorn、重新加载 Nginx、访问健康接口并核对提交号。

这次最关键的事故是：Gunicorn 通过 `/etc/tunerhub.env` 使用 PostgreSQL，而手动执行迁移时没有加载该文件，Django 自动改用了 SQLite。命令显示“没有迁移”，但检查的并不是生产数据库，最终新代码访问 PostgreSQL 缺少的字段并返回 `500`，让前端看起来像文章消失了。

以后执行任何生产迁移前，都必须先加载与系统服务相同的环境变量，并打印数据库引擎进行确认，但不能打印数据库密码。

## 六、故障排查方法

不要一看到页面异常就改代码。先建立一个能稳定复现问题的检查，再逐层比较：

1. 公网地址是否正常；
2. Nginx 是否正常；
3. Gunicorn 直连是否正常；
4. Django 中间件完整请求是否正常；
5. 直接调用视图是否正常；
6. 在生产环境变量下查询数据库是否正常。

例如，首页接口返回 `500` 不代表文章已被删除；它只说明前端没有拿到列表。直接视图返回 `200`、Gunicorn 返回 `500` 时，应优先比较服务进程和命令行的环境，而不是修改文章数据。

## 七、以后建站的完成标准

一个功能只有同时满足以下条件才算完成：

- 使用真实路由，刷新后仍能打开；
- 数据真实写入生产数据库；
- 普通用户、作者和管理员权限正确；
- 上传、加载、空状态、错误和成功反馈完整；
- 桌面端和手机端都能正常使用；
- 自动测试与前端正式构建通过；
- 数据库迁移和静态文件已经部署；
- 公网健康检查返回成功；
- GitHub 与服务器提交号一致。

这段经历最值得保留的底层原则是：永远把网站当作真实用户正在使用、真实数据正在积累、真实故障会影响信任的产品来建设。
